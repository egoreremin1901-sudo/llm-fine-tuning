# Dialogue Summarization with Qwen2.5 and LoRA

Проект посвящён задаче автоматической суммаризации диалогов. Основная цель — сравнить несколько подходов к генерации краткого summary для диалога и оценить, насколько дообучение модели через LoRA улучшает качество результатов.

В рамках проекта сравниваются три подхода:

- простой rule-based baseline;
- базовая LLM без дообучения;
- LLM после LoRA fine-tuning.

Для оценки используются автоматические метрики ROUGE и BLEU, а также дополнительная оценка через LLM-as-a-judge.

## Описание экспериментов

В проекте используются два основных скрипта.

### Первый скрипт

Первый скрипт отвечает за проверку базовых подходов без дообучения модели.

Сначала используется простой baseline: из каждого диалога берётся первое и последнее предложение, после чего из них формируется краткое summary. Такой подход нужен как минимальная точка сравнения. Он не использует нейросетевую модель, но позволяет понять, насколько другие методы лучше простого правила.

После этого используется базовая модель без дополнительного обучения. Модель получает диалог и генерирует summary. Затем для полученных ответов считаются метрики ROUGE и BLEU. Этот эксперимент показывает, насколько хорошо предобученная instruction-модель справляется с задачей без fine-tuning на конкретном датасете.

### Второй скрипт

Второй скрипт отвечает за дообучение модели с помощью LoRA.

В качестве основной модели используется `Qwen/Qwen2.5-0.5B-Instruct`. Дообучается не вся модель целиком, а только LoRA-адаптеры. Такой подход позволяет снизить требования к видеопамяти и ускорить обучение, что удобно для экспериментов в ограниченной среде, например Google Colab.

После fine-tuning модель применяется к тестовой выборке. Для сгенерированных summary считаются те же метрики ROUGE и BLEU, чтобы сравнить результат с baseline и с базовой моделью без дообучения.

## Почему выбрана Qwen/Qwen2.5-0.5B-Instruct

Модель `Qwen/Qwen2.5-0.5B-Instruct` выбрана из-за баланса между качеством и доступностью для запуска в ограниченной среде. Это небольшая instruction-модель, которую можно использовать в Google Colab без слишком больших требований к GPU-памяти.

Более крупные модели потенциально могут давать лучшее качество, но требуют больше ресурсов и дольше обучаются. Для учебного эксперимента модель размера 0.5B подходит лучше: она достаточно лёгкая для fine-tuning, поддерживает работу с инструкциями и совместима с LoRA/QLoRA-подходом.

## Метрики

Для оценки качества summary используются следующие метрики:

- `ROUGE-1` — оценивает совпадение отдельных слов между предсказанием и эталонным summary.
- `ROUGE-2` — оценивает совпадение пар слов.
- `ROUGE-L` — учитывает самую длинную общую подпоследовательность между текстами.
- `BLEU` — оценивает совпадение n-грамм между сгенерированным и эталонным текстом.

Эти метрики полезны для базового сравнения, но они не всегда полностью отражают смысловое качество summary. Хорошее summary может быть сформулировано другими словами и получить не самые высокие значения ROUGE или BLEU. Поэтому дополнительно используется LLM-as-a-judge.

## LLM-as-a-judge

LLM-as-a-judge используется как дополнительный способ оценки качества summary. Модель-судья получает:

- исходный диалог;
- эталонное summary;
- summary, сгенерированное оцениваемой моделью.

После этого модель-судья выставляет оценки по нескольким критериям:

- `faithfulness` — соответствие фактам из исходного диалога;
- `relevance` — отражение важной информации;
- `completeness` — полнота покрытия основных моментов;
- `fluency` — грамотность и читаемость текста;
- `overall` — общая оценка качества.

Такой подход помогает дополнить автоматические метрики и лучше оценить смысловую сторону сгенерированных summary.

## Общий pipeline

Общий порядок работы проекта:

1. Загрузка и подготовка датасета.
2. Разделение данных на train и test.
3. Создание rule-based baseline из первого и последнего предложения диалога.
4. Подсчёт метрик для baseline.
5. Генерация summary с помощью базовой модели без дообучения.
6. Подсчёт метрик для базовой модели.
7. Fine-tuning модели через LoRA.
8. Генерация summary с помощью дообученной модели.
9. Подсчёт метрик для LoRA-модели.
10. Дополнительная оценка результатов через LLM-as-a-judge.
11. Сохранение результатов экспериментов в таблицы.

## Возможные улучшения

Проект можно улучшить следующими способами:

- Использовать более крупную модель, например Qwen2.5 1.5B или 3B, если доступно больше GPU-памяти.
- Увеличить размер обучающей выборки.
- Подобрать более оптимальные параметры обучения: learning rate, batch size, количество эпох, LoRA rank.
- Добавить дополнительные проверки датасета: дубликаты, пустые значения, слишком длинные диалоги.
- Сравнить разные prompt-шаблоны для генерации summary.
- Использовать несколько моделей-судей для LLM-as-a-judge и сравнить их оценки.
- Добавить ручную проверку части примеров.
- Использовать дополнительные метрики, например BERTScore или semantic similarity через embeddings.
- Сохранять результаты всех экспериментов в единую таблицу для более удобного сравнения.

## Вывод

Проект демонстрирует базовый pipeline для задачи dialogue summarization: подготовка данных, создание baseline, генерация ответов базовой моделью, fine-tuning через LoRA, подсчёт метрик и дополнительная оценка через LLM-as-a-judge.

Такой подход позволяет сравнить простое правило, предобученную модель и дообученную модель, а также оценить, даёт ли fine-tuning улучшение качества на выбранном датасете.
